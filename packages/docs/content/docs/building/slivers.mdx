A **sliver** is a scrollable region with custom layout behavior. Flutter composes a single scroll view out of many slivers: a collapsing app bar, then a grid, then a list, then a footer — all sharing one scroll offset. Fuse exposes the whole suite as wrapper components you import from `solid-fuse`.

If you just need a scrolling column, reach for [`ScrollView`](/docs/building/ui-and-layout#scrollview) — it's simpler. Slivers are for when the scroll view itself needs structure: headers that pin or collapse, multiple lists with different layouts, grids, or pull-to-refresh.

## The host: CustomScrollView

Every sliver lives inside a `CustomScrollView`. Its children **are** slivers — you can't put a `View` or `Text` directly inside it; wrap leaf content in a sliver like `SliverToBoxAdapter` or `SliverList`.

```tsx
import { CustomScrollView, SliverAppBar, SliverList, Text, View } from "solid-fuse";

<CustomScrollView>
  <SliverAppBar title={<Text>Inbox</Text>} expandedHeight={200} pinned />
  <SliverList>
    <For each={messages()}>
      {(m) => (
        <View padding={16}>
          <Text>{m.subject}</Text>
        </View>
      )}
    </For>
  </SliverList>
</CustomScrollView>
```

`CustomScrollView` mirrors [`ScrollView`](/docs/building/ui-and-layout#scrollview)'s props — same physics, same controller, same axis switches.

| Prop | Type | Description |
|---|---|---|
| `scrollDirection` | `string` | `"vertical"` (default) or `"horizontal"` |
| `reverse` | `boolean` | Reverse scroll direction |
| `controller` | `ScrollController` | Programmatic scroll control (see [Handles](/docs/building/handles)) |
| `primary` | `boolean` | Use the inherited primary scroll controller |
| `physics` | `string` | `"bouncing"`, `"clamping"`, `"always"`, `"never"`, `"page"` |
| `shrinkWrap` | `boolean` | Size to content instead of filling the viewport |
| `anchor` | `number` | 0–1 — where the zero scroll offset sits in the viewport |
| `cacheExtent` | `number` | Pixels of off-screen content to keep built |
| `clipBehavior` | `string` | `"hardEdge"`, `"antiAlias"`, `"none"` |
| `keyboardDismissBehavior` | `string` | `"manual"` or `"onDrag"` |
| `dragStartBehavior` | `string` | `"start"` or `"down"` |
| `hitTestBehavior` | `string` | Hit-test behavior for the viewport |
| `restorationId` | `string` | Scroll-position restoration key |
| `paintOrder` | `string` | `"firstIsTop"` or `"lastIsTop"` — which sliver paints over the others. Needs a recent Flutter SDK; safely ignored on older versions. |

## How lists render: eager children, lazy layout

This is the model to keep in your head. **Your children render eagerly** — every `<For>` iteration produces a Solid node up front, the same as anywhere else in Fuse. But Flutter only *lays out and paints the visible window*. A `SliverList` of 10,000 rows builds 10,000 light Solid nodes, while Flutter inflates Flutter widgets for the dozen rows on screen (plus a small cache margin) and recycles them as you scroll.

Concretely: each sliver wraps your eager `children` in a lazy `SliverChildBuilderDelegate`. Children are keyed by their node identity, so inserting, removing, or reordering preserves widget state and scroll position instead of rebuilding everything below the change.

The practical consequences:

- **You don't manage windowing.** Render the whole list; Flutter handles what's visible.
- **The eager nodes still cost something.** 10,000 Solid nodes is fine; 10,000,000 is not. For unbounded data, **cap N at the data layer** — paginate or virtualize the array you pass to `<For>`, not the sliver. The sliver makes *layout* lazy, not your data.
- **Keys come for free.** Identity is tracked per node, so you don't add `key=` yourself.

## Lists and grids

### SliverList

Children are rows. By default rows are measured individually. If every row is the same height, set `itemExtent` — Flutter skips per-row measurement and scrolling gets cheaper. If rows share a size but you'd rather describe it with a widget, pass a `prototypeItem` instead.

```tsx
import { CustomScrollView, SliverList, Text, View } from "solid-fuse";

<CustomScrollView>
  <SliverList itemExtent={56}>
    <For each={contacts()}>
      {(c) => (
        <View padding={{ horizontal: 16 }} flex={{ justify: "center" }}>
          <Text>{c.name}</Text>
        </View>
      )}
    </For>
  </SliverList>
</CustomScrollView>
```

| Prop | Type | Description |
|---|---|---|
| `itemExtent` | `number` | Fixed row height (or width when horizontal). Cheapest option for uniform rows. |
| `prototypeItem` | `JSX` | A sample row whose measured size fixes every row's extent. Alternative to `itemExtent`. |
| `addAutomaticKeepAlives` | `boolean` | Keep off-screen rows alive (default `true`) |
| `addRepaintBoundaries` | `boolean` | Wrap each row in a repaint boundary (default `true`) |
| `addSemanticIndexes` | `boolean` | Add semantic index metadata (default `true`) |

### SliverGrid

Children are cells. Choose **one** of two layouts: a fixed column count (`crossAxisCount`) or a maximum cell width (`maxCrossAxisExtent`, columns computed to fit). Set one or the other, not both.

```tsx
import { CustomScrollView, SliverGrid, Text, View } from "solid-fuse";

<CustomScrollView>
  <SliverGrid crossAxisCount={3} mainAxisSpacing={8} crossAxisSpacing={8}>
    <For each={photos()}>
      {(p) => <View decoration={{ color: p.color, borderRadius: 8 }} />}
    </For>
  </SliverGrid>
</CustomScrollView>
```

| Prop | Type | Description |
|---|---|---|
| `crossAxisCount` | `number` | Fixed number of columns. Mutually exclusive with `maxCrossAxisExtent`. |
| `maxCrossAxisExtent` | `number` | Max cell width; column count is derived. Mutually exclusive with `crossAxisCount`. |
| `mainAxisSpacing` | `number` | Gap between rows |
| `crossAxisSpacing` | `number` | Gap between columns |
| `childAspectRatio` | `number` | Cell width-to-height ratio (default `1`) |
| `mainAxisExtent` | `number` | Fixed cell height along the scroll axis |
| `addAutomaticKeepAlives` / `addRepaintBoundaries` / `addSemanticIndexes` | `boolean` | Keep-alive flags, as on `SliverList` |

## App bars and collapsing headers

### SliverAppBar

A Material app bar that scrolls, collapses, pins, floats, and stretches. Set `expandedHeight` and a `flexibleSpace` for the classic large-photo-collapses-to-toolbar effect.

```tsx
import { FlexibleSpaceBar, Icon, SliverAppBar, Text, View } from "solid-fuse";
import { share, favorite } from "solid-fuse/icons/material";

<SliverAppBar
  pinned
  expandedHeight={240}
  title={<Text color="white">Mount Rainier</Text>}
  actions={[
    <Icon data={share} color="white" />,
    <Icon data={favorite} color="white" />,
  ]}
  flexibleSpace={
    <FlexibleSpaceBar
      background={
        <View
          width={Infinity}
          height={Infinity}
          decoration={{ image: { url: "rainier.jpg", fit: "cover" } }}
        />
      }
    />
  }
/>
```

Collapse behavior is fully declarative — `pinned`, `floating`, `snap`, and `stretch` describe what the bar does as you scroll; there's no shrink-offset callback to wire up. Use `type` for the Material 3 sizes (`"medium"` and `"large"` build the taller variants).

#### actions is an array of nodes

`actions` takes a JSX **array**, not children — `actions={[<A />, <B />]}`. This is the array-of-nodes prop pattern: each element is an independent JSX node serialized into the prop, laid out in the bar's trailing slot. The same shape applies anywhere a slot holds a list of widgets rather than a single child.

| Slot | Type | Description |
|---|---|---|
| `leading` | `JSX` | Widget before the title (back button, menu) |
| `title` | `JSX` | Title widget |
| `flexibleSpace` | `JSX` | Expanded-area content, usually a `FlexibleSpaceBar` |
| `bottom` | `JSX` | Persistent area below the toolbar (tabs, a search field) |
| `actions` | `JSX[]` | Trailing action widgets — an array of nodes |

| Prop | Type | Description |
|---|---|---|
| `type` | `string` | `"small"` (default), `"medium"`, `"large"` |
| `pinned` | `boolean` | Stays at the top when fully collapsed |
| `floating` | `boolean` | Reappears on any upward scroll |
| `snap` | `boolean` | With `floating`, snaps fully open/closed (no partial state) |
| `stretch` | `boolean` | Over-scrolls past `expandedHeight` |
| `forceElevated` | `boolean` | Always show the elevation shadow |
| `primary` / `centerTitle` / `automaticallyImplyLeading` / `excludeHeaderSemantics` | `boolean` | Standard app-bar flags |
| `expandedHeight` | `number` | Height when fully expanded |
| `collapsedHeight` | `number` | Height when collapsed (above the toolbar) |
| `toolbarHeight` | `number` | Toolbar row height |
| `elevation` / `scrolledUnderElevation` | `number` | Shadow depth, resting and scrolled-under |
| `titleSpacing` / `leadingWidth` / `bottomOpacity` | `number` | Layout tuning |
| `stretchTriggerOffset` | `number` | Over-scroll distance that fires `onStretchTrigger` |
| `backgroundColor` / `foregroundColor` / `shadowColor` / `surfaceTintColor` | `ColorInput` | Colors |
| `systemOverlayStyle` | `string` | `"light"` or `"dark"` status-bar icons |
| `clipBehavior` | `string` | Clip behavior |
| `onStretchTrigger` | `() => Promise<void>` | Awaitable callback fired when stretched past the trigger |

### FlexibleSpaceBar

Goes in `SliverAppBar`'s `flexibleSpace`. Holds a `title` and a `background` that animate as the bar collapses.

```tsx
import { FlexibleSpaceBar, Text, View } from "solid-fuse";

<FlexibleSpaceBar
  title={<Text color="white">Gallery</Text>}
  centerTitle
  collapseMode="parallax"
  stretchModes={["zoomBackground", "fadeTitle"]}
  background={
    <View
      width={Infinity}
      height={Infinity}
      decoration={{ image: { url: "cover.jpg", fit: "cover" } }}
    />
  }
/>
```

| Prop | Type | Description |
|---|---|---|
| `title` | `JSX` | Title widget, scales between expanded and collapsed |
| `background` | `JSX` | Background widget behind the title |
| `centerTitle` | `boolean` | Center the collapsed title |
| `titlePadding` | `EdgeInsetsInput` | Padding around the title |
| `expandedTitleScale` | `number` | Title scale factor when expanded |
| `collapseMode` | `string` | `"none"`, `"pin"`, `"parallax"` — how the background moves |
| `stretchModes` | `string[]` | Any of `"zoomBackground"`, `"blurBackground"`, `"fadeTitle"` |

## Sticky headers

These keep a header visible while a section scrolls. Pick by behavior:

| Component | Behavior | Notes |
|---|---|---|
| `PinnedHeaderSliver` | Sticks at the top, no resize | Simplest. Flutter ≥ 3.24. |
| `SliverResizingHeader` | Sticks and shrinks between two extents | Flutter ≥ 3.24. |
| `SliverFloatingHeader` | Hidden until you scroll up, then floats in | Flutter ≥ 3.24. |
| `SliverPersistentHeader` | Full control via a builder that gets the shrink offset | Builder/signal pattern; see below. |

### PinnedHeaderSliver

A single child that pins to the leading edge.

```tsx
import { PinnedHeaderSliver, Text, View } from "solid-fuse";

<PinnedHeaderSliver>
  <View padding={12} decoration={{ color: "white" }}>
    <Text fontWeight="bold">A</Text>
  </View>
</PinnedHeaderSliver>
```

### SliverResizingHeader

Pins and resizes between `minExtent` and `maxExtent` as you scroll.

```tsx
import { SliverResizingHeader, Text, View } from "solid-fuse";

<SliverResizingHeader minExtent={56} maxExtent={120}>
  <View flex={{ justify: "center" }} padding={{ horizontal: 16 }}>
    <Text fontSize={20}>Header</Text>
  </View>
</SliverResizingHeader>
```

| Prop | Type | Description |
|---|---|---|
| `minExtent` | `number` | Collapsed height |
| `maxExtent` | `number` | Expanded height |

### SliverFloatingHeader

Stays hidden until you scroll back toward it, then animates in over the content.

```tsx
import { SliverFloatingHeader, Text, View } from "solid-fuse";

<SliverFloatingHeader
  snapMode="overlay"
  animationStyle={{ duration: 200, curve: "easeOut" }}
>
  <View padding={12} decoration={{ color: "white" }}>
    <Text>Floating toolbar</Text>
  </View>
</SliverFloatingHeader>
```

| Prop | Type | Description |
|---|---|---|
| `animationStyle` | `{ duration: number; curve: string }` | Show/hide animation (duration in ms) |
| `snapMode` | `string` | `"overlay"` or `"scroll"`. Needs Flutter ≥ 3.27. |

### SliverPersistentHeader

The general-purpose header. Its child is a **builder** — a function that receives the current shrink offset and whether it overlaps scrolled content, and returns JSX. Use it when you want to animate a header yourself based on how far it's collapsed.

```tsx
import { SliverPersistentHeader, Text, View } from "solid-fuse";

<SliverPersistentHeader pinned minExtent={56} maxExtent={160}>
  {(shrinkOffset, overlapsContent) => {
    const t = 1 - shrinkOffset / 160; // 1 expanded → 0 collapsed
    return (
      <View
        decoration={{ color: "white" }}
        flex={{ justify: "center" }}
        padding={{ horizontal: 16 }}
      >
        <Text fontSize={20 + t * 12}>Scaling title</Text>
      </View>
    );
  }}
</SliverPersistentHeader>
```

| Prop | Type | Description |
|---|---|---|
| `pinned` | `boolean` | Stick at the top when collapsed |
| `floating` | `boolean` | Reappear on upward scroll |
| `minExtent` | `number` | Collapsed extent |
| `maxExtent` | `number` | Expanded extent |
| `children` | `(shrinkOffset, overlapsContent) => JSX` | Builder for the header content |

#### The builder/signal pattern and its ~1-frame lag

`SliverPersistentHeader` (and [`SliverLayoutBuilder`](#sliverlayoutbuilder) below) take a builder instead of plain children because the value they depend on — the shrink offset, the layout constraints — only exists *during Flutter's layout pass*, on the Dart side. There's no prop you could set ahead of time.

Under the hood the wrapper holds a Solid signal. Each Flutter layout pass pushes the live value across the bridge; the wrapper writes it into the signal, your builder re-runs reactively, and the new JSX flows back to Dart for the next frame. Because that round-trip is asynchronous, **the rendered child reflects the previous frame's value — a lag of roughly one frame.** For smooth collapse animations that's invisible. It does mean these builders aren't the place for layout that must be pixel-exact on the very first frame; prefer the declarative headers (`SliverResizingHeader`, `SliverAppBar`) when you can express the behavior without a builder.

## Layout and grouping

These slivers arrange or pad other slivers. Most take a single sliver child; the group slivers take many.

```tsx
import {
  CustomScrollView,
  SliverMainAxisGroup,
  SliverPadding,
  SliverPersistentHeader,
  SliverList,
} from "solid-fuse";

<CustomScrollView>
  {/* Pins each section's header above its own list */}
  <SliverMainAxisGroup>
    <SliverPersistentHeader pinned minExtent={40} maxExtent={40}>
      {() => <SectionHeader title="Today" />}
    </SliverPersistentHeader>
    <SliverPadding padding={{ horizontal: 16 }}>
      <SliverList>{/* rows */}</SliverList>
    </SliverPadding>
  </SliverMainAxisGroup>
</CustomScrollView>
```

| Component | Children/slot | Key props |
|---|---|---|
| `SliverToBoxAdapter` | single box child | Drops one ordinary widget into a sliver list |
| `SliverPadding` | single sliver child | `padding: EdgeInsetsInput` |
| `SliverFillRemaining` | single box child | `hasScrollBody` (default `true`), `fillOverscroll` (default `false`) — fills the leftover viewport |
| `SliverFillViewport` | children (cells) | `viewportFraction` (default `1`), `padEnds` (default `true`) — each child fills a viewport page |
| `SliverMainAxisGroup` | slivers | Stacks child slivers along the scroll axis; headers pin within the group |
| `SliverCrossAxisGroup` | slivers | Lays child slivers out across the cross axis |
| `SliverConstrainedCrossAxis` | single sliver child | `maxExtent: number` — caps the cross-axis size |
| `SliverCrossAxisExpanded` | single sliver child | `flex: number` — must be a direct child of `SliverCrossAxisGroup` |

### SliverLayoutBuilder

A **builder** that receives the live `SliverConstraints` and returns a sliver — the sliver analogue of a layout builder. Same builder/signal mechanism (and the same [~1-frame lag](#the-buildersignal-pattern-and-its-1-frame-lag)) as `SliverPersistentHeader`.

```tsx
import { SliverLayoutBuilder, SliverList, SliverGrid } from "solid-fuse";

<SliverLayoutBuilder>
  {(c) =>
    c.crossAxisExtent > 600 ? (
      <SliverGrid crossAxisCount={3}>{/* cells */}</SliverGrid>
    ) : (
      <SliverList>{/* rows */}</SliverList>
    )
  }
</SliverLayoutBuilder>
```

The constraints object exposes `crossAxisExtent`, `scrollOffset`, `remainingPaintExtent`, `viewportMainAxisExtent`, `precedingScrollExtent`, `overlap`, `axisDirection`, `growthDirection`, and `userScrollDirection`.

## Decoration and effects

Wrap a single sliver child to decorate or modulate it without breaking lazy layout.

| Component | Wraps | Key props |
|---|---|---|
| `DecoratedSliver` | a sliver | `decoration: DecorationInput`, `position` (`"background"` or `"foreground"`) |
| `SliverOpacity` | a sliver | `opacity: number`, `alwaysIncludeSemantics?` |
| `SliverAnimatedOpacity` | a sliver | `opacity: number`, `duration: number` (ms), `curve: string`, `onEnd?` |
| `SliverIgnorePointer` | a sliver | `ignoring: boolean` |
| `SliverSafeArea` | a sliver | `top` / `bottom` / `left` / `right` (default `true`), `minimum: EdgeInsetsInput` |

```tsx
import { DecoratedSliver, SliverList } from "solid-fuse";

<DecoratedSliver decoration={{ color: "white", borderRadius: 12 }}>
  <SliverList>{/* rows painted on the decoration */}</SliverList>
</DecoratedSliver>
```

## Reorderable lists

`SliverReorderableList` lets the user drag rows to reorder. Rows are eager children, exactly like `SliverList`. You provide `onReorder` and update your own data when it fires.

```tsx
import { createSignal, For } from "solid-js";
import {
  CustomScrollView,
  Icon,
  ReorderableDragStartListener,
  SliverReorderableList,
  Text,
  View,
} from "solid-fuse";
import { dragHandle } from "solid-fuse/icons/material";

function Playlist() {
  const [tracks, setTracks] = createSignal(["Intro", "Verse", "Chorus"]);

  const reorder = (oldIndex: number, newIndex: number) =>
    setTracks((t) => {
      const next = [...t];
      const [moved] = next.splice(oldIndex, 1);
      // Flutter reports newIndex as if the moved item is still present
      next.splice(newIndex > oldIndex ? newIndex - 1 : newIndex, 0, moved);
      return next;
    });

  return (
    <CustomScrollView>
      <SliverReorderableList onReorder={reorder} itemExtent={56}>
        <For each={tracks()}>
          {(track, i) => (
            <ReorderableDragStartListener index={i()}>
              <View padding={16} flex={{ direction: "horizontal", justify: "spaceBetween" }}>
                <Text>{track}</Text>
                <Icon data={dragHandle} />
              </View>
            </ReorderableDragStartListener>
          )}
        </For>
      </SliverReorderableList>
    </CustomScrollView>
  );
}
```

A row only becomes draggable from inside a `ReorderableDragStartListener` (drag starts immediately) or `ReorderableDelayedDragStartListener` (drag starts after a long-press). Pass each its row `index`.

| Component | Props |
|---|---|
| `SliverReorderableList` | `onReorder: (oldIndex, newIndex) => void` (required), `onReorderStart?`, `onReorderEnd?`, `itemExtent?`, `prototypeItem?` |
| `ReorderableDragStartListener` | `index: number`, `enabled?`, single child |
| `ReorderableDelayedDragStartListener` | `index: number`, `enabled?`, single child (long-press to start) |

## Pull-to-refresh

`onRefresh` is **awaitable**. You return a promise; the spinner stays up until it resolves. This is the one place where a callback's return value crosses the bridge and is awaited on the Dart side, so your refresh logic and the indicator animation stay in sync.

### RefreshIndicator

The Material spinner. Wrap a `CustomScrollView` (or any scrollable) in it.

```tsx
import { CustomScrollView, RefreshIndicator, SliverList } from "solid-fuse";

<RefreshIndicator
  onRefresh={async () => {
    const fresh = await api.fetchFeed();
    setItems(fresh);
  }}
>
  <CustomScrollView>
    <SliverList>{/* rows */}</SliverList>
  </CustomScrollView>
</RefreshIndicator>
```

| Prop | Type | Description |
|---|---|---|
| `onRefresh` | `() => Promise<void>` | Awaited; the spinner shows until it resolves |
| `triggerMode` | `string` | `"onEdge"` or `"anywhere"` |
| `color` / `backgroundColor` | `ColorInput` | Spinner and backdrop colors |
| `displacement` / `edgeOffset` | `number` | Spinner position |
| `strokeWidth` | `number` | Spinner stroke width |
| `adaptive` | `boolean` | Use the platform-native indicator (Cupertino on iOS) |

### CupertinoSliverRefreshControl

The iOS-style refresh control. Unlike `RefreshIndicator` it **is** a sliver — place it as the first child of a `CustomScrollView`, above your content.

```tsx
import { CupertinoSliverRefreshControl, CustomScrollView, SliverList } from "solid-fuse";

<CustomScrollView physics="bouncing">
  <CupertinoSliverRefreshControl
    onRefresh={async () => {
      await api.refresh();
    }}
  />
  <SliverList>{/* rows */}</SliverList>
</CustomScrollView>
```

| Prop | Type | Description |
|---|---|---|
| `onRefresh` | `() => Promise<void>` | Awaited; the control spins until it resolves |
| `refreshTriggerPullDistance` | `number` | Pull distance that triggers a refresh |
| `refreshIndicatorExtent` | `number` | Height held open while refreshing |

## Advanced: nested scroll views

`NestedScrollView` coordinates an outer header (slivers that collapse) with an inner scrollable body — the pattern behind a collapsing app bar that sits above a `TabBar` with independently scrolling tabs. Its `header` is a **builder** receiving whether the inner body has scrolled.

```tsx
import { NestedScrollView, SliverAppBar, Text } from "solid-fuse";

<NestedScrollView
  header={(innerBoxIsScrolled) => (
    <SliverAppBar pinned forceElevated={innerBoxIsScrolled} title={<Text>Feed</Text>} />
  )}
  body={<TabbedContent />}
/>
```

For overlap handling between the header and body, `SliverOverlapAbsorber` and `SliverOverlapInjector` resolve the absorber handle from the enclosing `NestedScrollView` automatically — you don't thread it through props.

| Prop | Type | Description |
|---|---|---|
| `header` | `(innerBoxIsScrolled) => slivers` | Builder for the outer header slivers |
| `body` | `JSX` | The inner scrollable body |
| `controller` | `ScrollController` | Outer scroll controller |
| `scrollDirection` | `string` | `"vertical"` or `"horizontal"` |
| `reverse` | `boolean` | Reverse scroll direction |
| `floatHeaderSlivers` | `boolean` | Let floating headers in the body float the outer header |

## Version gates

A few slivers depend on newer Flutter SDKs. They compile away or no-op on older versions, but to use them you need at least the version noted:

| Feature | Minimum Flutter |
|---|---|
| `PinnedHeaderSliver`, `SliverResizingHeader`, `SliverFloatingHeader` | 3.24 |
| `SliverFloatingHeader`'s `snapMode` | 3.27 |
| `CustomScrollView`'s `paintOrder` | recent SDK (silently ignored if unsupported) |

## What's not here, and why

Some Flutter slivers are deliberately omitted because they don't fit Fuse's reactive, eager-children model:

- **`SliverVariedExtentList` / `itemExtentBuilder`** — these need a synchronous per-index size callback, which the async JS↔Dart bridge can't provide. Use `itemExtent` (uniform) or let rows self-measure.
- **`SliverAnimatedList` / `SliverAnimatedGrid`** — their imperative insert/remove animation builders fight reactive eager children. Render with `<For>` and animate items with [`SliverAnimatedOpacity`](#decoration-and-effects) or per-row transitions until a reactive animation primitive lands.
- **`SliverFadeTransition`** — needs an `Animation` handle Fuse doesn't expose yet. Reach for `SliverAnimatedOpacity`.
- **`SliverOffstage` / `SliverVisibility`** — use Solid's `<Show>` to add or remove a sliver instead.
- **`AppBar` (non-sliver)** — it isn't a sliver and needs a `Scaffold`; it isn't part of this suite.
- **The Cupertino nav bar `.search` variant** — version-gated, omitted for now.
- **Low-level `Viewport` / `Scrollable` and raw sliver delegates** — out of scope; the wrappers above cover the practical surface.
